Web App Development

A web application is any client-server program where the intended client is a web browser.

In app development, a distinction is typically made between frontend (i.e. client or browser) and backend (i.e. server) development. Frontend and backend code is typically written in different languages, and developers often specialize in one or the other.

HTML

HTML is the markup language (not a programming language) used to define the structure of web pages rendered by browsers.

A HTML document is a (serialized) rooted tree, where all internal nodes are elements — each with an optional list of attributes — and leaves are either text or elements. There are a few different tags, and each tag supports a few different attributes. Browsers interpret these tags and attributes to load and display resources (or run them, in the case of JavaScript code).

You should familiarize yourself with commonly used elements and attributes, as well as how they’re used by the browser. A good way to build an intuition for this is to look at the HTML for a few random sites in your browser’s developer console (open a website, right click anywhere, then select ‘Inspect’/‘Inspect Element’).

CSS

CSS is the style sheet language (also not a programming language) used to define the appearance of web pages rendered by browsers.

CSS documents are paired with an HTML document (via <link rel="stylesheet" ...>...</link> or <style>...</style> tags in the HTML document) to tell browsers how the elements in the document should be displayed. A CSS document is a list of rules, where each rule consists of a selector (specifying which elements it applies to) and a list of properties (key-value pairs describing specific aspects of how elements should be display).

The CSS standard is fairly complex, and there are a ton of properties, so trying to learn all of it is pretty much futile. To further complicate things, new properties are constantly being added, and there are always lots of properties that aren’t yet supported in all browsers.

Still, everybody should have a basic familiarity with how rules work and some of the more common properties. A good way to get started is to look at and play around with CSS rules for a few random sites in your browser’s developer console (open a website, right click anywhere, then select ‘Inspect’/‘Inspect Element’).

Because CSS can get pretty verbose, more expressive languages that compile to CSS are sometimes used. The most commonly is called Sass/SCSS.

JavaScript

Generally considered a poorly designed language, JavaScript (JS) is nonetheless the de facto standard (and only option) for running code in browsers. It is an interpreted language, parsed and executed by a ‘JavaScript engine’ bundled with the browser.

JavaScript code in the browser typically interacts with one or more Browser APIs (also called ‘Web APIs’, ‘Device APIs’, etc.) and libraries (such as JQuery or React). The most fundamental browser API is the DOM API, which is used to interact with (read and modify) the tree defined by the current page’s HTML.

Unlike with CSS and HTML, trying to learn JavaScript by inspecting websites in the wild is hard, because most websites 1. run a ton of JavaScript (so it’s hard to make sense of it all), and 2. obfuscate (minify) their JavaScript code to the point of illegibility. A better way is to open up the console in your browser’s developer tools and try running commands.

Traditionally, websites have been server-side applications: sites where the client loads a fully rendered HTML page from the server, and occasionally runs JS snippets to perform various actions (i.e. the HTML is rendered server-side). Our focus, however, is on building client-side applications: sites where where the entire HTML page is built on the fly in the browser using JS (i.e. the HTML is rendered client-side). We use a library called React to accomplish this.

New versions of JavaScript are released yearly, and confusingly have two names (both named after the standard specification, called ECMAScript). The most significant recent release is ES6/ECMAScript 2015, which is generally well supported in modern browsers.

It’s becoming increasingly common to write frontend code in languages that compile to JavaScript, such as Typescript or Clojure/ClojureScript (which we use). One up-and-coming alternative to compiling to JavaScript is to compile to WebAssembly. WebAssembly is an assembly-like language supported directly by browsers, that promises greater performance and support for threads. It is fairly new, and not yet widely used or supported.

Things tend to move pretty fast in the JavaScript world, and staying on top of things is basically a full-time job. Unless you’re a JS developer, having a basic familiarity with the syntax and basic concepts should suffice.

Web Servers

A web server is any process (or hardware running any such process) that responds HTTP requests. The requests sent to web servers are typically instructions to create, read, update, or delete some resource (this set of operations is so common it has its own acronym: CRUD).

Web servers serve static files (i.e. plain old files), templates (HTML code that is dynamically generated), and/or web APIs (interfaces for performing operations).

Commonly served static files include HTML and CSS documents, JavaScript code, images (such as JPEG, PNG, or SVG files) and fonts (such as WOFF files). These days, static files are typically served via Content Delivery Networks (CDNs) instead of directly from a web server. This is done for performance and stability reasons (CDNs are distributed over the globe, usually provide Distributed Denial-of-Service (DDoS) protection, etc.).

Services

A service is any backend process that’s deployed as part of a web app. Apps consist of one or more services, most of which typically act as a web server. In addition to acting as web servers, services commonly perform tasks like communicating with databases, sending requests to other web servers, performing computations, etc.

Apps that consist of a single service are called monoliths. Traditionally most web applications were monoliths, but in recent years there’s been a trend towards Service-oriented architectures with an emphasis on microservices. Microservices are collections of lightweight services designed to one or a few things. We tend to favor microservices, but use a “monolith-first” approach (whereby we build larger services that we split up into smaller ones over time — this helps us avoid drawing the wrong boundaries for our microservices).

The code that runs in a web server can be written in any language – common choices include Python, Java, .NET, Ruby, and JavaScript (using Node.js). To perform whatever business logic they’re meant to perform, web servers typically either include a host of libraries (for e.g. talking to databases, rendering HTML documents, handling authentication etc.) or utilize web frameworks (such as Django for Python, Spring for Java, Ruby on Rails for Ruby, and Express for Node.js). Web frameworks are essentially collections of libraries ‘pre-wired’ for building web apps such that developers essentially ‘fill out the blanks’ – “you call libraries, but frameworks call you”.

The main benefit of frameworks compared to libraries is that they act as a ‘lingua franca’ and impose a certain rigidity upon the code, which can facilitate getting new developers up to speed with a project. Frameworks also make it easy (for developers who know them) to create prototypes and get started new applications.

On the other hand, framework code can often seem ‘magical’ and hard to understand (because the framework does almost all the work for you), and it can often be difficult or awkward to accomplish things not thought of or targeted by the framework developers. Swapping out or modifying components in a frameworks can also be hard or impossible. For these reasons, we tend to avoid frameworks in favor of libraries (or at least stick to microframeworks, which are frameworks that try to be a bit more lean/flexible).

Communication

There are tons of ways for backend services to communicate, but some are more common than others.

When it comes to web APIs the most common method is to use JSON (JavaScript Object Notation – a simplified JavaScript inspired syntax) over HTTP. XML and YAML over HTTP are also used in some contexts.

For service-to-service communications, JSON over HTTP is still common, but other data formats and protocols are often encountered as well. For instance, binary formats (like MessagePack, protobuf, and Avro) are often used to save space, bandwidth, and/or time. Message queues (like Kafka) are commonly used in place of HTTP (especially in microservice architectures) because they enable services to communicate asynchronously.

APIs

A significant portion of backend work is typically spent working on and with HTTP APIs.

Modern APIs typically use JSON for data transfer, and more often than not follow the REST ‘architecture style’ (at least to some degree; developers are often confused about what REST really is). Such APIs are called RESTful. REST is basically a set of guidelines for how to develop HTTP APIs such that they’re understandable and maintainable. The core focus is on resources and how they’re organized (i.e. how paths are associated with resources) and managed (i.e. how CRUD and other operations work). There’s usually one endpoint per action and resource; e.g. POST /api/images for posting a resource, GET /api/images for listing the resource, GET /api/images/<image-id> for retrieving a resource, and DELETE /api/images/:image-id for deleting a resource (sometimes GET /foo and POST /foo are considered to be the same endpoint, but accessed with different methods).

Rest APIs are typically static, i.e. they can’t easily be changed once people have started using them. When an API has to be modified, it’s common to just release a new version in parallel with the old, the idea being that the old one will be retired once all users have moved to to the new one (which is typically never). Multiple API versions are typically handled via separate path prefixes (e.g. /api/v1, /api/v2, et cetera).

Example — a minimal REST API using the Flask framework:

from flask import Flask, request, jsonify, abort
app = Flask('my-app')

values = {}

@app.route('/values', methods=['GET'])
def list_values():
    return jsonify(list(values.keys()))

@app.route('/values/<value_id>', methods=['GET', 'PUT', 'DELETE'])
def get_or_modify_value(value_id):
    if request.method == 'PUT':
        values[value_id] = request.get_json()
        return "", 200
    elif request.method == 'GET':
        if value_id in values:
            return jsonify(values[value_id])
        else:
            return "", 404
    else:
        if value_id in values:
            del values[value_id]
            return "", 200
        else:
            return "", 404

if __name__ == '__main__':
    app.run()

This app has a single resource (‘values’) which can be listed, read, created/updated, and deleted via the four endpoints (GET /values, GET/PUT/DELETE /values/<value_id>).

An alternative to REST that’s been gaining traction recently is GraphQL, which utilizes a single endpoint to respond to queries (sort of like SQL).
