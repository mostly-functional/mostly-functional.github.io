Programming Languages

Programming languages are formal languages used to write programs.

The concepts presented in this section can be useful for everyone, but knowing the details is essential mostly to people in engineering roles.

This section consists of two parts. In the first part, we discuss various programming language concepts. In the second part, we discuss the languages that we use; what they are, and how to learn to use them effectively.

Concepts

Computers execute machine code — a series of instructions written in a compact, binary format to be run on a specific processor architecture. Since writing machine code by hand is an extremely tedious and error-prone process, programmers (almost) always write code in a higher-level language — one that can be read and written by humans (as collections of text files, called source code). This code is then compiled (rewritten) to machine code by a program called a compiler.

Interpreted Languages

Some languages aren’t compiled directly to machine code — rather, they’re interpreted (read and executed on the fly) by a program called an interpreter (sometimes the code is compiled to an intermediate representation before being interpreted — e.g. in Java). This enables platform independence (code written in compiled languages have to be compiled once for each target platform, but in interpreted languages the interpreter handles any platform-specific details) and makes various language features easier to implement (e.g. dynamic typing — see below). Notable examples of interpreted languages include Java, Python, and JavaScript.

Interpreted languages are sometimes loosely referred to as scripting languages, especially when they’re used to create scripts (small programs that automate some task, usually contained within a single source code file).

Syntax, Semantics and Compilation

Programming languages (like regular natural languages) have grammars — rules that specify what is considered valid code (or a valid sentence), and rules that specify what the meaning any given piece of code (or sentence) is. The rules that define what is considered valid code in a language is called a language’s syntax, while the rules that specify what the code means are called its semantics.

Compilers typically work in multiple stages. The first step is called parsing. This is the process of reading source code, validating that it adheres to the language’s syntax, and mapping it to a data structure called an abstract syntax tree (AST). The compiler analyzes the AST to verify that the code is free of semantic problems, performs optimization (making the program faster without changing its semantics), and finally outputs machine code for the target platform as an executable program.

Interpreters typically perform all the above steps just like compilers, but they execute the code on the fly instead of outputting it as an executable. Some compilers output code in another language) instead of machine code.

Low-level and High-level Languages

Programming languages are often organized from low-level (more like machine code) to high-level (more abstract and expressive).

Low-level languages are used when working ‘close to the hardware’, and where performance is very important (e.g. in embedded systems, graphics programming, or simulations). The ‘lowest-level’ family of languages in practical use is called assembly languages — these are essentially just textual representations of machine code — i.e. a sequence of instructions geared towards some specific instruction set. Other popular low-level languages include C and Rust.

Low-level programming typically involves manually allocating/freeing and manipulating memory (using pointers). Writing such code is a somewhat tedious and error-prone process, which tends to lead to hard-to-detect security and stability problems (like buffer overflows, dangling pointers/segmentation faults, and memory leaks).

High-level languages are preferred in applications where developer productivity is more important than performance (which includes the vast majority of web apps). We mostly use the high-level languages Python and Clojure (which are both interpreted).

Memory in high-level (and especially dynamically typed) languages is usually garbage-collected. This means that the interpreter handles memory management so that the programmer doesn’t have to worry about it. Almost all web development happens in garbage-collected languages.

As an illustration of high- vs low-level programming, consider the example of reversing a string in C and in Clojure.

Types and Type Systems

Most/all programs revolve around repeatedly performing computations involving various values to produce other values. To help structure and reason about these computations, it’s very useful to distinguish categorize values according to what data they contain and what operations can be performed on them.

Programming languages handle this by organizing values into types. The set of rules that govern which types are permitted and how types interact is called a type system — every language has one of these. The type system’s rules are enforced by the compiler or interpreter. If this happens at compile-time (during compilation) it’s called static typing, and when it happens at run-time (when the program is being executed) it’s called dynamic typing.

Type systems also vary in how strict they are. Some languages (like C and JavaScript) allow types to be somewhat arbitrarily treated as other types — this is called weak typing and is generally considered a bad thing. A type system that is not weak is called strong.

State and Functional vs Imperative Languages

Apart from computing things, programs also need to keep track of things. The set of information kept track of by a program or other system is called its state. The process of modifying state is called mutation; values that can be mutated are called mutable and values that can’t be mutated are called immutable. Programming languages differ greatly in how they deal with state and mutation.

Most software is written using languages that take an imperative approach to state. In imperative languages, programs consist of sequences of statements, most of which typically modify the state referred to to by some variable (such as foo += 10 or bar[x] = bar[x + 1] + bar[x + 2]). Typically, most values in imperative languages are mutable, although some imperative languages (like C or Java) permit marking some variables (typically called constants) as immutable.

One popular class of imperative languages is object-oriented (OO) languages. In these languages, variables typically don’t refer directly to state, but to objects, which are a entities that hold one or more variables (typically called fields or member variables), as well as a set of actions (called methods) that can be called to read or mutate these variables. In most OO languages (Java, C++, etc.), objects are instances of classes, which are essentially custom types that specify a set of fields and methods.

Functional languages take a different approach to state. In these languages, all or almost all state is immutable, and the program state evolves by applying functions to existing values in order to produce new values. Languages in which all state is immutable are called purely functional. In these languages, all side effects (mutation, printing values, etc.) are captured by the type system and treated as data.

Functional, imperative, and object-oriented programming are said to be different programming paradigms. Imperative and functional programming can be viewed as different takes on what programs are: in the imperative approach, programs are treated as sequences that modify a set of values (or objects) until the desired state is achieved, whereas in the functional paradigm, programs are treated as a series of transformations which process data until the desired value is produced. Imperative programming is closer to how CPUs and memory work, so it can be more suitable for e.g. low-level programming. Functional programming — on the other hand — is more mathematical in its nature (there’s no mutation in mathematics), and as such can be easier to reason about, prove correct, parallelize, etc.

Most modern languages support a mix of paradigms, and let the programmer choose whether to write code in an imperative or functional style. In practice, such languages tend to lose a lot of the benefits of functional languages, because they don’t have referential transparency (essentially the property that function calls won’t have unexpected side effects).

Concurrency

Another important aspect of programming languages is how they deal with concurrency. This is when multiple parts of a program run concurrently — i.e. either in parallel at the same time (e.g. as different threads scheduled to different CPU cores), or interleaved (e.g. as different threads taking turn to run on a single-cored CPU).

Writing correct concurrent code when multiple threads of execution share mutable state is very hard, because operations from different threads can interfere with one another as well as happen in lots of different orders — these types of problems are called race conditions. Race conditions are traditionally avoided by means of synchronization mechanism (such as locks), but these can in turn cause other problems (such as deadlock).

Modern languages tend to work around these problems by avoiding shared mutable state in favor of other methods of communication, such as:

  • The actor model: threads can only communicate by sending messages asynchronously (i.e. messages can be written and read independently) to one another (used in Erlang and Scala).
  • Coroutines: threads are lightweight and communicate asynchronously via queues called channels (used in Go).
  • Software transactional memory: threads can share mutable state, but the system ensures that operations appear atomic to other threads (used in Clojure).

There’s also async/await, which is a special syntax for cleanly running asynchronous tasks (lightweight, temporary threads of execution) that return results (used in modern JavaScript).

As CPU frequencies have stagnated in recent years (and as the number of cores in a typical CPU have increased), concurrency support has become an increasingly important aspect of programming languages. As a result, modern languages tend to have a strong focus on concurrency.

It should be noted that functional languages generally have a strong edge when it comes to concurrency (compared to imperative languages), because their focus on immutable data means that there’s less potentially shared mutable state to worry about.

FFI and Scripting Languages

It’s fairly common occurrence for code written in one language to call code written in another, and most languages have one or more foreign function interfaces (FFIs), that facilitate this. Since OS and system APIs are typically written in C, FFIs targeting C are by far the most common (most languages have them). Some languages and also support embedding code written in other languages directly in the code (for instance, Assembly in C and C++).

It’s also common for languages to be embedded in applications or other systems. Examples include JavaScript in browsers (and some other applications), Visual Basic in Microsoft Excel, Bash in Unix, etc. These languages, when embedded in and used to control an application or other system, are called Scripting Languages.

Package Managers

Most modern applications are heavily reliant on libraries — both open source and ‘internal’ (i.e. shared within an organization). Manually downloading and including libraries in projects is a major pain — especially when said libraries need to be kept up to date, and have dependencies of their own — which is why all modern languages have a package manager.

Package managers typically handle downloading and caching of dependencies, conflict resolution, building and publishing libraries, etc. Some languages have more mature tooling for packages than others, but regardless, it’s important that you familiarize yourself with the package managers for the languages you develop in.

Noteworthy Programming Languages

Here are some languages in that are in common use or that we think are otherwise especially noteworthy:

  • Assembly Languages: Family of languages — essentially machine code in text format. Sometimes embedded in other low-level languages for performance reasons.
  • Bash: The most common Unix shell and scripting language. Not well-suited for programming, but a necessary evil when working on the command line.
  • C: Statically typed, imperative language used mostly for systems programming these days. Every operating system kernel and pretty much all standard Unix tools are written in C.
  • C#: Statically typed, object-oriented, and garbage-collected. Essentially a more modern Java. Used mainly by Windows programmers.
  • C++: C, but object-oriented, and with a crazy amount of other features tacked on. Used to be very popular, but these days it’s mostly used for game engines and other high-performance applications. Hard to do right, especially when working in teams.
  • Clojure: Dynamically typed, functional, and garbage-collected Lisp dialect with a focus on safe concurrency. Used mainly for backend and frontend development. A dialect of Lisp.
  • Go: Statically typed, garbage-collected, compiled, object-oriented language, with a focus on safe concurrency. Similar to C.
  • Haskell: Statically typed, purely functional language, with a powerful type system. More popular in academia than in industry, and a major inspiration for most modern languages.
  • Java: Statically typed, object-oriented, garbage-collected language. Mainly used for enterprise applications.
  • JavaScript: Dynamically typed, object oriented language. Although generally considered poorly designed (for one, it’s weakly typed), it’s widely used for both frontend and backend development, as well as a scripting language. JavaScript is the only language that runs in most browsers.
  • Julia: A modern, dynamically typed imperative language — essentially a more modern and faster Python. Mainly targets scientific applications. Basically a Lisp dialect, but with a more conventional syntax.
  • Lisp: Family of languages, usually dynamically typed, mostly functional (but often with some imperative features) languages. Notable for their extremely simple syntax (Lisps use a prefix notation and the code is written as a data structure). Pioneered many concepts that are ubiquitous today (dynamic typing, garbage collection, higher-order functions, etc.).
  • Lua: Simple, dynamically typed imperative language. Mainly used as a scripting language.
  • PHP: Scripting language mostly used for legacy backend applications. Widely considered poorly designed.
  • Python: Dynamically typed, optionally object-oriented imperative language, with a focus on simplicity. Widely used for backend development, and a lingua franca for data science.
  • R: Dynamically typed, garbage-collected language with a focus on statistical and scientific applications (via built-in support for linear algebra, plotting, etc.). Basically a Lisp dialect, but with a more conventional syntax.
  • Ruby: Dynamically typed, optionally object-oriented imperative language, with a focus on simplicity. Similar to Python. Mostly used for backend development.
  • Rust: Statically typed, modern alternative to C/C++, with a focus on safe memory management and concurrency. Its type system provides guarantees against common runtime errors encountered in other non-garbage-collected languages.
  • Scala: Statically typed, optionally functional language. Basically Java, but with Haskell glued on top. Runs on the Java Virtual Machine.
  • TypeScript: JavaScript but with optional static typing. Widely considered a ‘better JavaScript’, it’s gaining popularity for both backend and frontend development.

Python

Python is a dynamically typed language. It is widely used as an introductory language, as well as for scripting, service development, etc. in most areas of technology.

Learning Python is important and is likely a good investment of time for most people. We recommend that the reader select and work through an introductory text or course on Python.

Anybody who plans to do any serious development with Python should also be aware of the following (these, and more, are detailed):

  • The release of Python 3.0 broke backwards compatibility with previous versions, forcing people to rewrite their code to upgrade. Many packages and projects opted not to do this, which led to a bit of a split in the community for several years. As of 2020, Python 2 will no longer receive any updates. Any new project should use Python 3, and those working on Python 2 projects should use a tool like 2to3 to migrate their code to Python 3.
  • Python packaging is somewhat messy, and there are multiple different formats and tools in use. The standard package manager is pip, and the standard package repository is PyPi. Pip doesn’t natively support ‘per-project’ dependencies (it installs everything globally) — to handle this people use tools like venv and Pipenv.
  • Compared to many other languages, Python it is relatively slow and has poor support for concurrency (Python’s has a global interpreter lock (GIL) and prevents more than one thread from executing at the same time). It is possible to speed up parts of the code by using Python’s C FFI and tools like Cython.
  • Python has an official style guide called PEP 8. Python projects should follow this guide, and use a linter (like Pylama) to enforce it.
  • Knowing how to use a REPL such as iPython well is important for productivity.

Bash

Bash is the de-facto standard when it comes to scripting in Unix. It is both a shell (the standard shell in most Unix distributions) and a programming language.

Although most users think of the shell as an interactive command interpreter, it is really a programming language in which each statement runs a command. Because it must satisfy both the interactive and programming aspects of command execution, it is a strange language, shaped as much by history as by design. — Brian W. Kernighan & Rob Pike

While Bash isn’t great as a programming language, it’s still important to know it well because:

  1. Lots of tasks require working at the command line, and doing that well requires an understanding of Bash.
  2. In lots of situations, it’s the best (or only) choice for scripting.
  3. You’re bound to come across Bash scripts written by others fairly frequently.

The best way to learn Bash is probably to spend a little time up-front getting familiar with the syntax and learning basic commands and constructs, and then to try to learn continuously as you work on the command line.

Clojure

Clojure is a dynamically typed functional language, belonging to the Lisp family of languages.

Like all Lisps, its syntax is rather distinct, because it uses prefix notation rather than infix notation (e.g. (+ 1 2 3) instead of 1 + 2 + 3), and the code is written directly as a data structure (e.g. (print "hello" (* (+ 1 2) (- 3 4))) instead of print("hello", (1 + 2) * (3 - 4))). This syntactic simplicity can make code in Clojure and other Lisps feel unfamiliar, but it also enables a level of extensibility not found in other languages (because Lisp code is data and can be manipulated by the language itself — see homoiconicity).

We use Clojure because it makes it easy to write robust code quickly. It also allows us to write the backend and frontend in the same language: we use regular Clojure running on the Java Virtual Machine (JVM) (Java’s interpreter) on the backend, and ClojureScript (a Clojure compiler that targets JavaScript) on the frontend.

The best way to learn Clojure is to work with it interactively using a REPL – you should always use one when learning the language or writing code (see ‘REPLs’ below for information on how). When first getting into the language, you can ignore more advanced topics (like macros, transducers, protocols/records/types, multimethods, and clojure.spec).
