Computer Architecture and Operating Systems

While having a deep understanding of computer hardware and operating systems is not vital to most roles, having at least a superficial understanding is important for understanding many other concepts. In this section, we provide a overview of some of the hardware and operating system concepts most relevant to application development (and the rest of this document). The material is pretty fundamental, and anybody working in tech should be somewhat familiar with it.

CPUs

The central processing unit (CPU) (often called just ‘processor’) is the chip that executes the instructions that make up programs.

Any given CPU model understands a specific set of instructions, called its instruction set. There are lots of instruction set architectures, but in practice the only two that matter are x86 (which is what the CPU in your computer uses) and ARM (which is what the CPUs in your phone and tablet use).

The frequency at which a processor executes its instructions is called the clock rate. Advancements in computing power used to be driven by ever increasing CPU clock rates, but this increase has plateaued in recent years (due to physical limits).

These days, processors typically have a variable clock rate that maxes out somewhere in the 2-5 GHz range, and advancement in CPU computing power happens by squeezing more cores (processing units that can run independently of one another) onto the CPU chip. This increase in cores can only be utilized by software that’s been explicitly written to run using multiple threads.

Memory and Disk

Computers store data in two places: in memory (using random access memory (RAM) chips) and on disk (using some form of data storage, like hard disk drives (HDDs) or solid state drives (SSDs)).

Memory is volatile (it is erased when the computer is turned off), while disk storage is permanent, so disk is used for any data that needs to be stored long-term. On the other hand, writing to and reading from memory is much faster than to/from disk, so running applications tend to keep the data they need to operate in memory.

Personal devices usually have up to 32 GB of memory and up to 1 TB of disk, while servers can have up to multiple TBs of memory and hundreds of TBs of disk.

Other Components

The graphics processing unit (GPU) is a computer component primarily used to perform graphics-related computations. GPUs are built for data parallelism (performing one operation on many pieces of data at the same time) and typically have thousands of cores. Because of this, lately they’re increasingly being used for non-graphics-related tasks involving high data parallelism, such as training machine learning models or mining Bitcoin.

The motherboard is the component into which all the aforementioned components plug in, together with keyboards, USB devices, etc.

Binary, Encodings and Compression

All hardware devices operate on sequences of bits (a bit is a single base-2 number), usually organized into chunks of 8 called bytes. Data stored as bits is called binary (often written as a sequence of 1s and 0s, prefixed with 0b; e.g. 0b10110011). Binary data is commonly displayed in hexadecimal (hex) form (base-16 numbers; displayed using 0-9 followed by A-F or a-f; often prefixed with 0x; e.g. 0xF12A). One byte is two hexadecimal characters (e.g. 0b11011100 = 0xFC).

Since hardware operates on binary data, this means that any data that’s not binary by nature (e.g. text, audio, images, etc.) must be encoded to binary before it can be processed or stored.

For text, this happens via a character encoding. There are several character encodings in common use, but the most popular these days is UTF-8 (one of the encodings specified as part of the Unicode standard). UTF-8 is designed to be backwards-compatible with the older and much more limited ASCII standard (which is common in older systems, but which can’t encode most alphabets).

Embedding binary data where text is more convenient is usually done using a binary-to-text encoding such as Base64 (base-64 numbers; displayed using A-Z, followed by a-z, followed by 0-9, usually followed by + and /).

Non-textual data is typically stored using specialized encodings (like for audio, video, and images). These encodings typically employ some form of data compression in order to reduce the file size. Data compression methods can either be lossless (i.e. no information is lost when encoding the data) or lossy (i.e. the encoding process throws away some information present in the data in order to reduce file size). Most media you’ll encounter (such as JPEG images or MP3 audio) uses lossy compression.

Another area where lossless compression is used is for archive files. These are files that bundle together multiple files and directories (together with some metadata) and then compress them. Common such formats include ZIP (typically associated with the filename extension .zip) and tar (typically associated with the filename extensions .tar, .tar.gz, .tgz, .tar.xz, and .txz, depending on which method of compression is used).

Operating Systems

The software that handles communication with and management of a computer’s hardware is called a kernel. Kernel are bundled with various utilities, graphical interfaces, etc. as Operating Systems (OSs).

The dominant OSs are Windows, macOS (formerly OS X), and Linux. Linux isn’t really a single OS, but rather a family of OSs that all use the Linux kernel.

Linux and macOS both belong to the Unix family of OSs. This means that they’re very similar from a usage and application development perspective, and any given piece of software is typically available for both.

Kernels provide application programming interfaces (APIs) for programmers/programs to interact with. Calls made to these APIs are called system calls (or syscalls).

Kernel APIs are built around a set of abstractions, which help provide a more convenient interface than dealing directly with the underlying hardware. The following abstractions are especially important:

  • Processes: A process is an instance of a running program. Processes can make syscalls to launch (or spawn) other processes — these are called child processes of the original process. Each process has a unique Process ID (PID).
  • Threads: A thread is a sequence of instructions running independently as part of a program. Processes start with one thread, and can spawn more threads via syscalls. Used to make programs to multiple things at the same time.
  • Files: A file is a chunk of data, given a filename (sometimes containing a filename extension like .html) and stored on disk. Files can be created, moved, deleted, and opened (for reading and/or writing) through syscalls.
  • The file system: a big data structure into which all files on a computer are organized. In Unix, the file system is organized as a single rooted tree of directories (or folders); the top-level directory of which is called the root directory and denoted /. In Windows, the file system consists of multiple rooted trees (with root directories C:, D:, etc.). Note: Confusingly, the word ‘file system’ is also used to refer to the format used to serialize files on disk.
  • Sockets: Sockets are channels that can be opened either between processes running on the same machine (e.g. Unix sockets) or between hosts on different machines (network sockets) for the purpose of sending data back and forth.

The following abstractions are also good to be aware of:

  • Virtual memory: A chunk of memory allotted to a process to store data in. If all available memory is exhausted, the OS sometimes transparently relocates parts of the virtual memory to disk (this is called paging, and the place where the data is stored is called a swap file or swap partition).
  • Environment variables: A set of values, addressed by keys, used to pass configuration information to processes. Each process has an associated set of environment variables, which are usually ‘passed down’ to its child processes.
  • Signals: notifications sent from the OS to a process, typically to tell it to stop running.

In order to be executed by the OS, a program must first be compiled (translated to machine code using the appropriate instruction set) and stored as an executable binary file (in a format understood by the OS). Programs can delegate some work to shared libraries, which are chunks of code written by somebody else, that have been compiled and are stored somewhere on the file system.

Modern OSs are multi-user; they have multiple user accounts, often but not always associated with specific end users (humans). Each file, directory and process is owned by some user account (typically the account that created it), and users can not access files or directories owned by other users unless they have been granted permission to do so. There’s also a special user called the superuser (also called root user or administrator), which always has permission to do whatever it wants. OSs typically have mechanism that allow end users to execute specific commands as the superuser.

OSs usually ship with both a graphical user interface (GUI) and a command-line interface (CLI) called a [shell](https://en.wikipedia.org/wiki/Shell_(computing). Most people who use computers never interact with a shell, but knowing how to use a shell well is an essential skill for power users.
