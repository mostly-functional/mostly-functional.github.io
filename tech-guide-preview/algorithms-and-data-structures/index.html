Algorithms and Data Structures

Computer science students spend a lot of time building a deep understanding of algorithms and data structures. Luckily for you (unless you’re a computer scientist) you’ll most likely never need this knowledge in your day-to-day work. Nevertheless, it’s good to be aware of some foundational algorithms and data structures.

Graphs

While research on algorithms and data structures tends to get pretty theoretical and involve sophisiticated mathematical machinery, this machinery can usually be dispensed with in non-mathematical contexts (such as this document).

There is, however, one mathematical construct that occurs time and time again in applications: the graph.

Graphically, a graph is just a collection of dots, connected by lines or arrows, such as the following examples:

A)              | B)              | C)        | D)              | E)              | F)
                |                 |           |                 |                 |
 * --- * --- *  |  * --- * --- *  |  * --- *  |  * --> * --> *  |  * --> *     *  |   * <-- *
       |        |  |     |        |           |        |        |  |     |     ^  |   |     ^
       |        |  |     |        |           |        v        |  v     v     |  |   v     |
 * --- *        |  * --- *        |  * --- *  |  * <-- * --> *  |  * <-- * --> *  |   * --> *

Mathematically, a graph is defined as a collection (V, E) of vertices (or nodes — the dots) V = {v1, v2, v3, ...} and a collection of (distinct) edges (the lines/arrows) E = {e1, e2, ...} of pairs of vertices. Graphs ‘with lines’ are called undirected graphs, while graphs ‘with arrows’ are called directed graphs (in undirected graphs the edges are two-element sets, and in directed graphs they’re ordered pairs).

A path in a graph is any sequence of nodes where an edge exists between each subsequent pair of nodes (intuitively: where each ‘step’ of the path goes through an edge) — in a directed graph, paths must ‘follow the arrows’.

A graph that has no cycles (i.e. there’s no way to start at a vertex, follow edges without repetition, and end up back at the same vertex) is called acyclic. Directed acyclic graphs (DAGs) are commonly encountered in applications. Graphs A, C, D, and E are acyclic, and D and E are both DAGs.

A graph where there’s at least one path between all pairs of vertices is called connected. Graphs A, B, and E above are connected.

A tree is a connected acyclic undirected graph. Graph A above is a tree, but B and C aren’t (B has a cycle, and C is not connected).

A directed tree (confusingly, often just called tree) is a connected acyclic directed graph, where no more than one edge ‘points to’ each nodes. Graph D above is a directed tree, but E and F aren’t (in E, multiple edges point to the bottom left node, and F has a cycle). Directed trees are commonly encountered in applications. In a directed tree, the node that ‘points to’ a node is called its parent node, and the set of vertices it itself points to is called its child nodes. There’s always one node that doesn’t have a parent — this is called the root node of the tree (the upper left node in D). Nodes that don’t have children are called leaf nodes (the lower left, lower right, upper right nodes in D).

Data Structures

A data structure is a structure for holding collections of items. Any given data structure comes with a set of operations, like inserting, accessing, and removing items.

It’s good to be familiar with the following fundamental data structures:

  • Arrays: an array is a sequence of items — usually created with a fixed maximum size — where items can be inserted and accessed by their index.
  • Graphs: a data structure representing a graph, where each node holds an item.
  • Linked lists: a sequence of items, where each item holds a reference to the next (i.e. a type of graph).
  • Trees: a data structure representing a directed tree, where each node holds an item. Trees typically support insertion, deletion, membership checks, and searches (the tree structure is often optimized for this).
  • Sets: a set is an unordered collection of items, where there can be no duplicate elements.
  • Stacks: a stack is a sequence of items, where items are both inserted and read/removed at one end. The order elements are removed from a stack is called last in, first out (or LIFO).
  • Queues: a queue is a sequence of items, where items are both inserted at one end and read/removed at the other end. The order elements are removed from a queue is called first in, first out (or FIFO).
  • Associative arrays (also called records, maps, and dictionaries): a collection of key-value pairs, where there can be no duplicate keys. Key-value pairs can be inserted and modified, as well as deleted (via the key), and values can be retrieved (also via the key).

Algorithms

An algorithm is a sequence of steps that describes how to solve a problem. Algorithms take an input (problem instance) and produce an output (solution).

It’s good to be aware of the following algorithms:

  • Sorting algorithms: algorithms that put lists of objects in order.
  • Search algorithms: algorithms that find things, such as binary search for finding an element in a sorted list, and breadth-first search/depth-first search for finding elements in a graph.
  • Hash functions: algorithms that map arbitrary values to strings of fixed size, such that collisions (many inputs being mapped to the same output) are unlikely. Cryptographic hash functions like SHA-2 and bcrypt are commonly used for authentication.

Complexity

The computational complexity of an algorithm is a measure of how the number of operations it performs increases with the size of its inputs. It’s important because performance on large datasets tends to dominate in applications.

Computational complexity is usually described using big O notation; for any algorithm, this is (basically) the number of operations performed by the algorithm on an input containing n elements, in a worst-case scenario (i.e. for the slowest possible input), ignoring constant factors and lower-order terms (for instance, if an algorithm requires 50 + 2n + 3n² operations, it is O(n²)).

As a rule of thumb, an algorithm that looks at a fixed number of elements is O(1) (it has constant complexity), an algorithm that looks at each element in turn is O(n) (it has linear complexity), an algorithm that looks at every pair of elements is O(n²), and an algorithm that looks at every permutation of elements is O(n!) (factorial complexity — such algorithms basically take forever even for small number of elements).

For example the following algorithm — which computes the sum of a list of numbers — is O(n):

def sum_numbers(numbers):
    sum = 0
    for number in numbers:
        sum += number
    return sum

…whereas the following algorithm — which computes the sum of all pairs in a list of numbers — is O(n²):

def sum_number_pairs(numbers):
    sum = 0
    for i, number_1 in enumerate(numbers):
        for number_2 in numbers[i:]:
            sum += number_1 + number_2
    return sum

Different algorithms for solving the same problem often differ in complexity; for instance, the following algorithm does the same thing as the previous one, but is O(n):

def sum_number_pairs(numbers):
    sum = 0
    count = len(numbers)
    for number in numbers:
        sum += number
    return sum * count
